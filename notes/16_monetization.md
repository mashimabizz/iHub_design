## 2. 基本原則

ファン層に嫌われない**5つの原則**：

### 原則1：コア機能は永久無料

```
❌ 「以前は無料だった機能の有料化」（ベネフィット引き上げ）
✅ 「便利機能・装飾の有料化」
```

無料で使える機能：
- マッチング・検索・フィルタ・保存検索
- 在庫登録・ウィッシュ登録（数量制限なし）
- AW登録（実用的な数量）
- 打診送信・受信・ネゴ・合意・取引・C-3 全プロセス
- メッセージ送信・受信
- 通報・dispute申告
- 評価表示

### 原則2：取引フロー中は広告ゼロ

```
✅ 広告 OK：閲覧系画面（ホーム・検索・ウィッシュ等）
❌ 広告 NG：取引フロー（C-0〜C-3）、認証、設定、法的、ヘルプ
```

理由：取引体験を壊すと iHub の存在意義そのものが揺らぐ。

### 原則3：推し活関連の広告のみ

```
❌ 絶対NG：パチンコ、出会い系、消費者金融、FX、医療系怪しいやつ
✅ OK：公式グッズ、ライブチケット、推し活グッズショップ、コラボカフェ、聖地巡礼ツアー
```

### 原則4：個人情報を広告会社に渡さない

- マッチングデータ・取引履歴は内部のみ使用
- 広告ターゲティングは「推しグループ」レベルまで（個人を特定しない）
- プライバシーポリシーで明記

### 原則5：透明性

- 収益源を全公開
- スポンサー枠は「★ Sponsored」明示
- アフィリエイトリンクは明示
- ステマ撲滅

## 4. Phase別ロードマップ

### Phase α（MVP〜半年）：完全無料

> ユーザー獲得が最優先。マネタイズはゼロ。

- 広告なし
- ブーストなし
- Premium なし
- アフィリエイトなし
- 「応援する」ボタン（任意の投げ銭）のみ可能 ⚠️

**目標**：DAU 100-500、コア機能の完成度向上、初期コミュニティ形成

### Phase β（半年〜1年）：広告 + ブースト導入

> 主な収益源は広告とブースト。Premium はまだ。

- バナー広告 / Native広告 開始（Tier 1 + Tier 2）
- ブースト機能ローンチ（単発¥150〜、5個セット¥600）
- 開示：「広告は推し活関連のみ」をプライバシーポリシーで明記

**目標**：DAU 500-1,500、月収 ¥10K-50K

### Phase γ（1年〜2年）：Premium + アフィリエイト追加

> 「広告うざい人」と「ヘビーユーザー」向けに Premium 会員開放。

- Premium 会員ローンチ（月¥500 / 年¥4,800）
- アフィリエイト導入（Amazon / 楽天 / 公式チケット販売）
- 直接スポンサー営業開始（Phase β で実績作ってから）

**目標**：DAU 1,500-5,000、月収 ¥50K-200K

### Phase δ（2年〜）：公式コラボ + 法人化検討

> 規模化、安定収益。

- 公式イベント運営との提携（「TWICE 名古屋ドーム公式 x iHub」等）
- 直接スポンサー営業の本格化
- 法人化（合同会社）検討
- B2B SaaS 開発検討

**目標**：DAU 5,000+、月収 ¥200K+

## 5. 広告（Ad）

### 5-1. 配置方針

**Tier 1: Native ad（カードと同形）— 推奨**

「★ Sponsored」と明示しつつ、コンテンツ列に自然混入：

| 画面 | 配置 | 頻度 |
|---|---|---|
| `HOM-main`（ホーム） | マッチカード列 | 5枚毎に1枚 |
| `SCH-main`（検索結果） | 検索結果列 | 5件毎に1件 |
| `WSH-list`（ウィッシュ） | アイテム列 | 5件毎に1件（アフィリリンク優先） |

**Tier 2: フッターバナー — 控えめ**

スクロール最下部 or 固定下部：

| 画面 | 配置 |
|---|---|
| `PRO-other`（他人プロフ） | 下部に control 付きバナー |
| `TRD-past`（過去取引） | リスト下部 |
| `AW-list`（自分のAW一覧） | 下部に |

**Tier 3: 広告ゼロ画面（必ず守る）**

```
❌ 認証・オンボーディング全画面（AUTH-*, ONB-*）
❌ 打診作成（C0-*）
❌ 打診受信（C1-receive）
❌ ネゴチャット（C15-*）
❌ 取引チャット（C2-chat）
❌ 証跡撮影・承認・評価（C3-*）
❌ Dispute 全画面（D-*）
❌ 設定・プロフィール編集（SET-*, PRO-edit, PRO-oshi-edit 等）
❌ 法的画面・ヘルプ（LEG-*, HLP-*）
```

### 5-2. 広告の種類

| 種類 | サイズ | 使用場面 |
|---|---|---|
| Native（カード型） | コンテンツと同形 | Tier 1 |
| バナー | 320×100 / 320×50 | Tier 2 |
| 自社広告 | 任意 | iHub Premium 訴求、ブースト購入訴求 |

### 5-3. 広告ネットワーク

| Phase | ネットワーク | 特徴 |
|---|---|---|
| Phase β 開始 | **Google AdSense** または **AdMob**（モバイル） | 即時開始可、CPM ¥200-500 |
| DAU 5,000+ | **直接スポンサー営業** | 単価高（CPM ¥500-1,500）、推し活ブランド限定 |

### 5-4. 広告非表示条件

| ユーザー状態 | 広告表示 |
|---|---|
| 無料 + ブースト未発動 | 全 Tier 表示 |
| **無料 + ブースト発動中（24h）** | **全広告非表示**（24h 限定） |
| **Premium 会員** | **全広告非表示**（永続） |

### 5-5. 広告コンテンツポリシー

許可：
- 公式グッズ（ペンライト・トレカ・グッズ通販）
- ライブチケット販売
- 推し活ツアー（聖地巡礼・遠征宿）
- 推し活カフェ・コラボイベント
- 推し活アプリ（音楽・イラスト・SNS）
- 関連メディア（推し活マガジン・YouTube番組）

絶対禁止：
- パチンコ・スロット・カジノ
- 出会い系・マッチングアプリ
- 消費者金融・キャッシング
- FX・暗号資産
- 怪しい医療・サプリ
- 政治・宗教
- 競合（同種の交換アプリ）

⚠️ 要確認：許可ジャンルの境界線（成人向け同人誌販売など）

## 6. ブースト機能（Boost）

### 6-1. 概要

「自分の打診や在庫を、相手のマッチング画面で目立たせる」課金機能。

「**広告スキップ + 上位表示**」の2機能をパッケージ。Tinder の Boost や Bumble の Spotlight に類似。

### 6-2. 効果

ブースト発動中（24h）：

| 効果 | 内容 |
|---|---|
| **上位表示** | 相手のマッチング画面・検索結果でこのユーザーを上位に |
| **目立つバッジ** | 「★ ブースト中」バッジ表示 |
| **広告非表示** | このユーザー自身も広告を見なくなる |
| **「優先マッチ」通知** | 受信者にプッシュ通知（任意・控えめ） |

### 6-3. 価格モデル

| パッケージ | 価格 | 単価 |
|---|---|---|
| 単発1個 | ¥150 | ¥150/個 |
| 5個セット | ¥600 | ¥120/個 |
| 10個セット | ¥1,000 | ¥100/個 |
| Premium 会員特典 | 月3個無料配布 | 実質 ¥167/個 |

### 6-4. 発動制約

| 制約 | 値 |
|---|---|
| 1日の発動上限 | 2個/日（依存性防止） |
| 1ブーストの有効時間 | 24時間 |
| 残数表示 | 常時マイページに表示 |
| 未成年購入 | 不可（規約で年齢確認必須） |
| 払い戻し | 未使用分のみ可（30日以内） ⚠️ |

### 6-5. ブースト発動シーン

ユーザーが選べる：

1. **打診送信時**：「相手の通知で目立つ」（最も推奨される使い方）
2. **マッチ閲覧中**：「相手のマッチング画面で上位表示」
3. **取引チャット中**：「即時返信を促す」（軽い催促効果） ⚠️ 効果薄、要検証

### 6-6. UI / UX

「ブースト購入」画面（仮称：BST-purchase）：
- 残数表示
- 購入オプション（1/5/10）
- 「Premium ならお得」訴求
- 決済（Stripe / Apple/Google Pay）

「ブースト発動」UI：
- マッチング画面右上の「⚡ ブースト」ボタン
- 発動確認ダイアログ
- 発動後、24h カウントダウン表示

### 6-7. 関連データモデル

```
boosts table:
- id (uuid)
- user_id (uuid → users)
- granted_at (timestamptz)              # 取得時刻
- granted_via ('purchase' | 'premium_grant')
- consumed_at (timestamptz nullable)    # 発動時刻
- expires_at (timestamptz nullable)     # 発動から24h後
- transaction_id (uuid → transactions, nullable)

ad_overrides table:
- user_id (uuid → users)
- override_until (timestamptz)          # 広告非表示の終了時刻
- override_reason ('boost' | 'premium')
```

詳細は `notes/05_data_model.md` 参照。

## 7. Premium 会員

### 7-1. 設計原則

「**コア機能の差はゼロ、装飾と便利機能のみ**」を貫く。

ユーザー反感を最小化するため、Premium = 「広告 + 雑事を取り除く」と「装飾」のパッケージ。

### 7-2. 特典一覧

| 特典 | カテゴリ |
|---|---|
| **広告完全非表示** | 鬱陶しさ除去 |
| **月3ブースト無料配布** | 数量提供 |
| **メッセージ履歴：6ヶ月 → 永続保存** | 副次的 |
| **既読確認表示** | 副次的 |
| **プロフ背景・絵文字カスタマイズ** | 装飾 |
| **「★ Premium」バッジ** | 装飾（虚栄心） |
| **AW登録：実用的な数量 → 無制限** ⚠️ | 副次的（無料も実用範囲） |
| **検索条件保存：5個 → 20個** ⚠️ | 副次的 |

⚠️ コア機能（マッチング通知・打診・取引）は **完全無料**で差をつけない。

### 7-3. 価格

| プラン | 価格 | 月単価 |
|---|---|---|
| 月額 | ¥500 | ¥500 |
| 年額 | ¥4,800 | ¥400（20%お得） |

### 7-4. 解約・払い戻し

- いつでも解約可能（即時 or 期間終了時）
- 未使用月分の払い戻しは原則なし（年額契約）
- 解約後も期間終了まで使える

### 7-5. 関連データモデル

```
subscriptions table:
- id (uuid)
- user_id (uuid → users)
- plan_type ('monthly' | 'yearly')
- status ('active' | 'cancelled' | 'expired')
- started_at / current_period_end (timestamptz)
- cancelled_at (timestamptz nullable)
- transaction_provider ('stripe' | 'apple' | 'google')
- transaction_provider_id (string)
```

詳細は `notes/05_data_model.md` 参照。

| 連携先 | コミッション率目安 | 用途 |

### 8-3. 表示原則

- 「★ アフィリエイトリンク」と明示

- ユーザーがクリックしなければ収益ゼロ → 押し付けがましさゼロ

## 10. B2B SaaS（Post-MVP）

### 10-1. 構想

「推し活コミュニティ向け管理ツール」を別商品として展開：

- ライブ運営向け：会場周辺の交換動線管理
- グッズメーカー向け：販売後の二次流通動向（匿名集計）
- ファンコミュニティ向け：イベント企画・グッズ交換補助

### 10-2. ビジネスモデル

- 月額 ¥30K-300K（規模次第）
- カスタマイズ込みなら別途見積

### 10-3. 取り組み時期

- iHub MVP 完成後（2年目以降）
- 法人化と同時に展開推奨

## 11. 禁止事項

### 11-1. 絶対やらないこと

```
❌ コア機能の有料化（マッチング・打診・取引）
❌ 個人情報の販売
❌ パチンコ・出会い系・消費者金融広告
❌ ステルスマーケティング（「★ Sponsored」表示なし広告）
❌ ガチャ要素のあるブースト（ランダム性で煽る）
❌ 課金しないと使い物にならない設計
❌ 解約しづらいUX（dark pattern）
```

### 11-2. 後発で検討するもの（廃案候補）

検討したが採用しないもの：

- ❌ NFT / トークン化（規制リスク・ファン層に合わない）
- ❌ データ販売（プライバシー）
- ❌ ガチャ系課金（依存性）
- ❌ サブスクBOX（物販はiHubの本筋ではない）
