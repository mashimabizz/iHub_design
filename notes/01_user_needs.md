# ユーザー要求整理（随時更新）

## 起点となる仮説（依頼者より）

- 交換相手を見つけるのが大変
  - 自分の交換条件をSNSにもれなく書くのが大変
  - 逆に探すのも大変
  - 写真をいちいち撮るのが面倒

## 起点となる機会仮説（依頼者より）

- 個人対個人の1:1交換に閉じている → 多者間で組み合わせれば成立する交換が眠っている（※郵送多者間はMVP対象外）
- 140字に収まる「譲／求」リストでは、自分の全在庫の中の交換可能性を表現しきれない
- いま交換条件に書いていないものでも、相手にとっては欲しいものかもしれない

## 取引対象の理解

**交換対象は「ランダム封入の小型グッズ」が中心**：
- トレカ・生写真・缶バッジ・アクスタ・キーホルダー等
- 単品グッズ（うちわ・ペンライト・タオル等）は通常交換対象外
- ランダム封入＝だぶり大量発生＝交換需要の源泉

**結果として**：
- 単価帯はジャンル横断でほぼ同じ（数百円〜数千円）
- 物理シーンも横断で大差なし
- ジャンル別UI最適化の必要性は低い → **横断スタートで設計可**

## 戦略前提（詳細は 03_strategy.md）

- **Xは捨てない**：iHubはXへの導線
- **現地交換特化のMVP**：郵送はYahooフリマと棲み分け
- **1:1中心、3者まで（現地のみ）**
- **ジャンル横断スタート**（MVPの初期データはKPOPからスタート）
- **物理シーン非依存**

## ペインと要求（カテゴリ別）

### A. 出品（自分の持ち物を出す）周りのペイン

- A-1. **写真撮影が地獄**：傷確認・表裏・コーティング有無で時間が無限に溶ける（20枚で30分以上）
- A-2. **自分の在庫を頭で管理**：把握漏れ・二重DMが発生
- A-3. **140字に書ききれない潜在在庫**がある
- A-4. **「譲」も実は条件付き（相手依存）**
- A-5. **状態表現がバラバラ**：iHub内タグで統一できる
- A-6. **【現地特有】携帯グッズと自宅在庫の切り分けができない**

### B. 検索・マッチング周りのペイン

- B-1. **「求」の優先度が並列に見えてしまう**
- B-2. **ウィッシュリスト×相手出品の突合が手作業**
- B-3. **【現地特有】会場にいる／来る予定の人が見つけられない**
- B-4. **【現地特有】「いま開封したばっかり」のリアルタイム在庫マッチが回らない**

### C. 交渉・コミュニケーション周りのペイン

- C-1. **DMの取り回しがしんどい**
- C-2. **相手の信頼性確認が手作業**
- C-3. **【現地特有】時間プレッシャー**

### D. 物理交換・現地運用のペイン【最重要】

- **D-1. 本人特定が怖い／困難**
- **D-2. 物理交換の証跡が残らない【最重要】**
  - **解決方針**：両者の交換物を1枚の写真に収めて両者が承認する設計（透かし等不要、普通の写真アップでOK）
- **D-3. 会場の電波が悪い【最重要】**：オフラインファースト設計
- **D-4. 会場の物理ロジスティクス**

### E. 信頼・安全性周りのペイン

- E-1. **アカウント種別の判別が困難**
- E-2. **詐欺っぽいアカウントの判定材料が少ない**
- E-3. **未成年同士の現地交換の心理的安全性**

### F. 戦略的な機会領域

- F-1. **在庫一回登録で出品テキスト・写真が自動生成**
- F-2. ~~交換ポリシー設定~~ → **MVPでは廃止**
- F-3. **マッチ提案の自動化**
- F-4. **取引実績・評価がプロフに集約**
- F-5. **状態表現が標準化されたタグ**
- F-6. **「いまこの場所＋時間の交換マーケット」を俯瞰**

## ペルソナの理想シナリオ（現地交換フルフロー）

1. iHubに自宅在庫を登録（**初回1回だけ大変、ここのUXが命**）
2. 公演／イベント当日、「携帯グッズ」をタップして持参在庫だけに絞り込み
3. 「Xに投稿」ボタンで、譲求テキスト+写真を自動生成
4. Xに投稿、リプ・DMで反応
5. 「iHubで取引しませんか？」リンクで誘導
6. iHubで在庫・ウィッシュリストを相互照合
7. 場所・時間を合意、現地合流
8. **両者の交換物を1枚の写真に収めて相互承認**、物理交換、相互評価

## マッチング設計（確定）

### 在庫の公開モデル

**全部公開**：登録した在庫はすべて検索・マッチ対象。譲りたくない物は登録しない運用。

### マッチング 2軸構成

| 種別 | 条件 | UX |
|------|------|-----|
| **完全マッチ** | 私の携帯グッズ × 相手wish ＋ 私のwish × 相手譲 ＋ 場所・時間一致 | 最優先・即時通知 |
| **片方向マッチA：あなたを求めてる人** | 私の携帯グッズ × 相手wish 一致 | 提案・タブ表示 |
| **片方向マッチB：あなたが求めてる人** | 私のwish × 相手譲 一致 | 提案・タブ表示 |

### UIタブ構成

```
[ 完全マッチ ★★★ ] [ あなたを求めてる人 ] [ あなたが求めてる人 ] [ 探索 ]
```

### 片方向マッチ → 完全マッチへの昇格導線（採用）

- 片方向マッチで相手プロフを開くと、相手の譲一覧が見える
- 「私のwishは相手の譲には無いけど、相手の譲一覧から興味あるものを探す」ことで完全マッチに昇格
- 「書いてないけど引き出しに眠ってた交換可能性」を引き出す

### 打診プロセス（折衷案で確定）

1. 両者に半マッチ通知（※通知頻度制御はMVP外）
2. どちらかが「打診する」ボタン → 自動生成テンプレ＋編集 → メッセージ送信
3. 相手が諾否（OK / NO / ブロック）
4. OKでチャット開始 / NOで終了（ブロックではない）

### 補助機能

- **「断った」学習**：「この相手は断った」を片方向で記憶（重複打診防止）
- **NO時の定型文**：「すでに先約があります」「またの機会に」など選択肢を提供
- ※ 通知頻度制御・「もう持ってる」マークはMVP対象外
