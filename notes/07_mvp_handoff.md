# iHub MVP 仕様書（プロトタイプ用ハンドオフ）

（2026-04-27 確定 / 自己完結型仕様書）

このドキュメントだけでプロトタイプ生成に必要な情報が揃うよう自己完結型で構成。
Claude Design 等のデザインツールにそのまま渡せる。

---

## 1. プロダクト概要

iHub は、グッズ交換の **現地（オフライン）取引** に特化したプラットフォーム。

### 解くペイン
- 交換相手探しが大変（X140字の制約・写真撮影の負荷・DM管理の心理コスト）
- 自分の在庫を頭で管理している（把握漏れ・二重DM）
- 現地での合流・本人特定が怖い／困難
- 物理交換の証跡が残らずトラブル時に水掛け論
- 会場の電波が悪く既存サービスが動作しない
- 140字に書ききれない潜在在庫がある

### 棲み分け
- **Yahooフリマ**：郵送特化　→　iHubは現地特化
- **X**：呼びかけ・交渉のSNS　→　iHubはマッチング・取引の専用プラットフォーム
- **iHub × X**：双方向の導線（Xは「初期接触」、iHubは「マッチング・取引・取引履歴」）

### 長期ビジョン
**「現地交換の最適化ツール」に集中、機能は絞る**

---

## 2. ターゲットユーザー

10代〜40代の女性ヲタ、月数回〜週数回のグッズ交換頻度。

代表ペルソナ「ハナ」：27歳・会社員、KPOP・邦アイのWヲタ、月3〜5回交換、ツアー期は週2〜3回。

---

## 3. 取引対象

ランダム封入の小型グッズが中心：
- トレカ・生写真・缶バッジ・アクスタ・キーホルダー等
- 単品グッズ（うちわ・ペンライト・タオル）は通常交換対象外
- 単価帯：数百円〜数千円
- ジャンル横断対応（**MVP は KPOP 限定**でスタート）

---

## 4. ユーザージャーニー（核）

1. **オンボーディング**：性別・主な活動エリア・「推し」（KPOPグループ／メンバー）登録
2. **在庫登録**：自宅在庫を登録（だぶりは ×N で一括、写真は手動切り抜き）
3. **ウィッシュリスト登録**：求めるグッズ（最優先・2番手・妥協OK の優先度＋曖昧度）
4. **公演・イベント当日**：「携帯グッズ」をタップで持参在庫に絞り込み
5. **Availability Window 設定**：時間範囲＋場所＋許容半径（既存イベントタグから一括入力可）
6. **マッチ確認**：4タブ（完全マッチ／あなたを求めてる人／あなたが求めてる人／探索）
7. **X投稿（任意）**：在庫・ウィッシュ選択 → 譲求テキスト＋画像を自動生成 → Xに投稿
8. **X→iHub 誘導**：Xからのリプ・DMを「iHubで取引」リンクで誘導
9. **打診プロセス**：
   - 完全マッチ：自動表示・即時通知
   - 片方向マッチ：両者通知 → どちらかが打診ボタン → 自動生成テンプレ＋編集 → 相手諾否（OK/NO/ブロック）
   - OK→チャット開始 / NO→終了（断り、ブロックではない）
10. **取引チャット**：場所・時間の合意（地図／会場ピン）
11. **現地合流**：服装写真の直前更新、QRコード提示
12. **物理交換**：両者の交換物を1枚の写真に収め、両者承認
13. **相互評価**：★・コメント

## 5. MVP 機能要件

### F1. ユーザー管理
- アカウント登録・プロフィール
- マッチ前公開：性別、主な活動エリア、評価サマリ（★平均・取引回数・無断キャンセル数）
- マッチ後公開：取引時の格好写真（直前更新可）、交換物詳細
- 通報・ブロック機能
- **「推し」登録**（パーソナライズ・通知の基盤）

### F2. 在庫管理（自宅在庫）
- 写真撮影（**手動切り抜き**）：複数グッズ1枚撮影 → 矩形ドラッグで切り抜き → 400×400 JPEG
- 入力項目：キャラ（運営マスタ参照）／グッズ種別（運営マスタ参照）／状態タグ／**数量×N**／メモ
- **「全部公開」モデル**：登録すれば全員に検索される（プライベート・条件付きはMVPなし）
- **「携帯グッズ」フラグ**：is_carrying ＋ carry_event_id でイベント別絞り込み

### F3. ウィッシュリスト
- キャラ／グッズ種別（NULL許可で曖昧マッチ）
- **flexibility**：exact / character_any / series_any
- **priority**：最優先・2番手・妥協OK

### F4. Availability Window（合流可能枠）
- 時間範囲・場所（lat/lng）・許容半径（500m / 1km / 駅エリア）
- イベントタグへの紐付け（任意のショートカット）

### F5. マッチング（4タブ構成）

| タブ | 内容 |
|------|------|
| **完全マッチ ★★★** | 私の携帯グッズ × 相手wish ＋ 私のwish × 相手譲 ＋ 場所・時間一致 → 即時通知 |
| **あなたを求めてる人** | 私の携帯グッズ × 相手wish 一致（私のwish側は未一致） |
| **あなたが求めてる人** | 私のwish × 相手譲 一致（私の譲側は未一致） |
| **探索** | 場所・時間・キャラ・グッズ種別で自由フィルタ |

- **昇格導線**：片方向マッチで相手プロフを開くと相手の譲一覧が見える → 「他の在庫から見つける」ことで完全マッチ昇格
- **「断った」学習**：片方向で記憶、重複打診を防止

### F6. 打診・コミュニケーション
- 打診ボタン → 自動生成テンプレ（編集可）→ 相手に送信
- 相手の諾否：OK（チャット開始）／ NO（定型文選択肢で優しく断り）／ ブロック
- 取引チャット（テキスト＋画像）
- 場所・時間の合意（地図／会場ピン）

### F7. 物理交換【最重要】
- **合流支援**：QRコード提示・服装写真直前更新・会場マップ・目印登録
- **証跡撮影**：両者の交換物を1枚の写真に収める（普通の写真アップ＋メタデータ：両者ID・タイムスタンプ・任意で位置）
- **両者承認**：両者の確認で取引完了、関連 user_haves.status を traded に
- **オフラインファースト**：在庫・ウィッシュ・既存マッチ・連絡先・合流場所はローカルキャッシュ。撮影証跡もオフライン撮影→復帰時アップロード

### F8. 評価・トラブル対応
- 取引後の相互評価（★1-5、コメント任意）
- サマリ集計：★平均、取引回数、無断キャンセル数
- **通報フロー**（証跡画像をエビデンスとして添付可）

### F9. X連携
- iHub→X：在庫・ウィッシュ選択 → 譲求テキスト＋写真を自動生成 → 投稿
- X→iHub：「iHubで取引する」誘導リンク

### F10. コレクション進捗
- 「シリーズ単位の所持数／総数」可視化
- 「コンプまであと N 枚」サジェスト
- 在庫データの集計のみ、UI は最低限

### F11. キャラ／グッズ種別運営マスタ
- 階層：genres_master → groups_master → characters_master
- aliases[] で表記揺れ吸収
- **MVPは KPOP 限定**：BTS／TWICE／NewJeans／IVE／Stray Kids など5〜10グループからスタート

### F12. イベントタグ
- ユーザー作成・サジェスト・名寄せ
- Availability Window への一括入力ショートカット
- 公式情報の自動取り込みは後フェーズ

## 6. データモデル（要点）

### 既存テーブル（変更なし）
- `users`
- `user_haves`（status: active/reserved/traded/withdrawn）
- `user_have_images`（複数画像対応の予備）
- `user_wants`（flexibility, priority）

### 既存テーブルへの変更
- `user_haves.is_carrying` boolean 追加
- `user_haves.carry_event_id` uuid nullable 追加（→ events）
- `user_haves.genre_tag_id` を必須化
- `user_haves.character_id` カラム追加（マスタ参照、character_name は表示キャッシュ）
- `user_haves.exchange_method` の "郵送" は MVP では UI 非表示（データ型は維持）
- `user_haves.hand_prefecture` を削除し、`users.primary_area` に格下げ移管
- 数量は **DBは1行=1個 を維持**、UI で「×N」を受け取って N行 INSERT

### 新規テーブル
- `genres_master`、`groups_master`、`characters_master`、`goods_types_master`
- `availability_windows`：時間範囲＋場所＋半径
- `events`：公演／物販イベントタグ
- `proposals`：打診（送信者・受信者・対象 haves/wants・status）
- `messages`：取引チャット
- `exchanges`：成立した取引（参加者・証跡写真URL・両者承認）
- `user_evaluations`：評価（★・コメント）
- `rejected_partners`：「断った」記録（重複打診防止）
- `user_oshi`：「推し」登録
- `reports`：通報

## 7. マッチング計算ロジック

### 既存ロジック
- genre_id 完全一致
- character_id 照合（want側 NULL なら無視）
- goods_type_id 照合（want側 NULL なら無視）
- exchange_method && 重複（MVPは "手渡し" のみ）

### 追加ロジック
- **場所・時間**：availability_windows の時空交差
- **携帯グッズ絞り込み**：完全マッチタブでは is_carrying = true のみ対象
- **片方向マッチ**：完全マッチで除外された候補を片方向タブに振り分け
- **「断った」フィルタ**：rejected_partners 既登録ペアを除外
- **flexibility 反映**：want側の flexibility に応じて character_id/goods_type_id をスキップ

### 計算タイミング
- バッチ：低頻度の定期計算
- オンデマンド：タブを開いたときに最新化
- リアルタイム：完全マッチ発見時のみ即時通知

## 9. 非機能要件

- **モバイルファースト**：iPhone・Android、片手操作前提
- **オフライン耐性【最重要】**：会場の電波が悪い前提で、最低限の機能はキャッシュで動作
- **時間プレッシャーへの応答性**：会場で1〜2分以内に交換成立まで進める
- **プライバシー**：本名不要、マッチ相手以外プロフ非表示徹底
- **アクセシビリティ**：人混み・屋外・夜間の操作性

---

## 10. MVP対象外（明示）

- 郵送取引（Yahooフリマと棲み分け）
- 多者間交換（4者以上、郵送3者）
- **同担拒否フラグ**
- イベントカレンダー（タグ機能のみ）
- 長期フォロー機能
- 海外対応（日本語限定）
- 詳細オンボーディング
- アクセシビリティ深掘り
- 通知頻度制御
- MobileSAM／背景白塗り／裏面対応
- 写真透かし／高解像度別系統

## 13. デザイン方針詳細

### ブランドガイド
- **メインカラー**：紫＋水色（パステル寄りのラベンダーパープル＋ソフトな水色）
- **アクセントカラー**：必要に応じて白／淡いピンク
- **トーン**：可愛い寄り ─ 丸みのある角・柔らかいタイポ・余白多め・パステル基調
- **対象年齢を踏まえた調整**：子供っぽくならず、10代〜40代の女性をまたいで使える落ち着き

### 既存プロトタイプとの関係
- **既存プロトタイプは無し**（コードレベルの試作はあるが、UIのリファレンスとしては使わない）
- デザインは Claude Design でゼロから生成する想定
